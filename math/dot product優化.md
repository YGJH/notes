### **第一部分：為何我的程式庫能比 PyTorch 快 20 倍？**

影片開頭，作者拋出了一個驚人的成果：在相同的計算任務上，他的程式庫比 PyTorch 快 20 倍。他對此也感到驚訝，並由此引出兩個問題：
1.  這怎麼可能？
2.  如何能做得更多，讓它變得更快？

為了解釋這個現象，作者首先對比了 **C 語言** 和 **Python** 的本質區別：

*   **C 語言（編譯式語言 Compile Language）**:
    *   程式碼需要先**編譯 (Compile)** 成 CPU 可以直接讀取的二進制檔案（binary file）。
    *   一旦編譯完成，這個二進制檔可以被重複執行，每次執行時 CPU 都能立即讀取並運行指令，速度非常快。

*   **Python 語言（直譯式語言 Interpreted Language）**:
    *   Python 的底層是由 C 語言寫成的程式。當你執行一個 Python 檔案時，這個 C 程式會打開你的 Python 檔，逐行解析語法（例如 `A+B`），將其翻譯成一種稱為「Python 字節碼 (bytecode)」的中間碼，然後再執行。
    *   這個過程**每次執行都需要重複一次**，因此有巨大的效能開銷。
    *   此外，Python 是**動態型別 (Dynamic Typing)**語言。當執行 `A+B` 時，解釋器需要進行大量檢查（例如：A 是不是字串？是不是數字？B 能不能跟 A 相加？），這些檢查都非常耗時。

**Python 的「秘密武器」：C API**
聰明的 Python 開發者知道 Python 速度慢，所以提供了一個 **C API**，讓開發者可以用 C 或 C++ 編寫高效能的擴充模組，然後再將它「包裝」成一個 Python 函數來使用。這就是像 **NumPy** 和 **PyTorch** 這類高效能 Python 程式庫的運作原理。

**PyTorch vs. 作者的程式庫**
儘管 PyTorch 使用了 C++ 後端，但它的架構是：
*   **PyTorch**: 大約 **60% 的程式碼是 Python**，剩下的是 C++。當你呼叫一個 PyTorch 函數時，程式會先在 Python 層執行一些操作（準備工作、檢查等），然後再進入 C++ 核心執行真正耗時的計算。這個從 Python 進入 C++ 的過程本身就帶來了**額外開銷 (Overhead)**。
*   **作者的程式庫**: **99% 的程式碼是 C++**。當你呼叫他的函式時，程式幾乎是直接進入已編譯好的 C++ 二進制碼，沒有 Python 解釋器介入的延遲和開銷。

**結論**:
影片開頭的「20 倍速」測試，是在一個**模型很小、輸入資料也很小**的情境下完成的。在這種情況下，計算本身不複雜，效能瓶頸主要來自於**呼叫函式的「額外開銷」**。由於 PyTorch 的 Python 層開銷較高，而作者的程式庫開銷極低，因此展現出了巨大的速度差異。

### **第二部分：瓶頸分析與新目標**

作者坦承，隨著模型和輸入資料量的增大，PyTorch 的效能會逐漸追上甚至超越他的程式庫。這是因為在大型計算中，「額外開銷」的佔比變小，而**核心計算（特別是矩陣的點積運算 Dot Product）的效率**成為了決定性因素，而 PyTorch 在這方面的優化非常出色。

因此，作者的新目標是：**優化他程式庫中的矩陣點積運算**，使其在大型計算上也能保持競爭力。

矩陣點積是一個 `O(n³)` 級別的複雜運算（涉及三層巢狀迴圈），是神經網路中最耗時的部分，也是優化的關鍵。

### **第三部分：矩陣點積的詳細優化過程**

作者展示了他如何一步步優化 C++ 中的矩陣點積函式，將其執行時間從 **10 秒**縮短到 **0.5 秒**（剛好也是 20 倍的提升）。

1.  **初始版本 (Baseline)**: 一個未經優化的 C++ 實現，執行時間為 **10 秒**。
    *   **問題**: 作者使用了自製的陣列類別 (Array Class)。每次透過 `array[i][j]` 存取資料時，都會觸發類別的建構子 (constructor)、解構子 (destructor) 和形狀檢查，產生大量不必要的函式呼叫和物件創建開銷。

2.  **優化 1：快速存取 (Fast Get)**
    *   **方法**: 建立一個 `fast_get` 函式，繞過所有類別的包裝，直接透過指標和簡單的數學運算（乘法和加法）從記憶體中讀取資料。
    *   **結果**: 執行時間從 10 秒降至 **6 秒**。

3.  **優化 2：快速設定與遞增 (Fast Set/Increment)**
    *   **方法**: 仿照 `fast_get` 的邏輯，創建 `fast_set` 和 `fast_increment` 函式，同樣用來直接操作記憶體。
    *   **結果**: 執行時間從 6 秒降至 **3 秒**。

4.  **優化 3：內聯化 (Inlining)**
    *   **方法**: 將 `fast_get` 等函式的程式碼直接寫入迴圈內部，消除函式呼叫本身的開銷（例如建立堆疊、移動暫存器等）。
    *   **結果**: 執行時間從 3 秒降至 **2.5 秒**。

5.  **優化 4：迴圈展開 (Loop Unrolling)**
    *   **核心思想**: CPU 在執行指令時，如果下一條指令的輸入依賴於上一條指令的輸出（例如 `output = output + value`），就必須等待。
    *   **方法**: 作者不使用單一的累加器 (accumulator) `output`，而是創建了 8 個獨立的累加器（`acc0`, `acc1`, ... `acc7`）。在迴圈的每次迭代中，同時計算 8 個值並分別加到這 8 個累加器中。因為這 8 個累加器之間互不依賴，CPU 可以並行處理這些加法運算。迴圈結束後，再將這 8 個累加器的總和加到最終的 `output` 中。
    *   **結果**: 這是最顯著的優化，執行時間從 2.5 秒大幅降至 **0.9 秒**。

6.  **優化 5：移動迴圈不相關的計算**
    *   **問題**: 作者發現在最內層的迴圈中，有一項計算其實只跟外兩層迴圈的變數有關，卻在最內層被重複計算了數百次。
    *   **方法**: 將這項計算移到它所依賴的迴圈層級，避免不必要的重複運算。
    *   **結果**: 執行時間從 0.9 秒降至 **0.5 秒**。

### **第四部分：與業界標準比較與未來展望**

經過一系列優化，作者的程式碼本身實現了 20 倍的效能提升。但這與真正的頂級效能相比如何？

作者將他優化後的函式與一個業界標準的線性代數庫 **BLAS (Basic Linear Algebra Subprograms)** 進行了比較。BLAS 是 PyTorch 等許多科學計算庫的底層核心。

*   **結果**: **BLAS 的點積運算比作者優化後的版本還要快約 30 倍**。

*   **原因**: BLAS 使用了更高級的優化技術，主要是：
    1.  **SIMD (Single Instruction, Multiple Data)**: 「單指令多資料流」，一種 CPU 指令集，可以讓一條指令同時對多個資料執行相同的操作，極大地提升了平行計算能力。
    2.  **多線程 (Multi-threading)**: 將計算任務分配到 CPU 的多個核心上同時執行。

**結論與下一步**
作者的旅程還未結束。他意識到要真正挑戰 PyTorch，就必須實現這些更高級的優化。影片的結尾，他預告下一部影片將會介紹如何在他的程式庫中加入 **SIMD** 和 **多線程** 優化。
