這四個是 Rust 語言中核心的 **Trait (特徵)**，專門用來處理 **「比較」** (Comparison) 的邏輯。

簡單來說，它們定義了程式語言中 `==`, `!=`, `<`, `>`, `<=`, `>=` 這些運算符的行為。

為什麼 Rust 要把它們拆得這麼細？主要是為了處理像 **浮點數 (Floating Point)** 這種「不乖」的數學邊界情況。

以下是詳細的拆解與比較：

---

### 1. 相等性比較 (Equality)

這兩個 Trait 負責處理 `==` 和 `!=`。

#### **`PartialEq` (部分相等)**

- **定義**：這是一個較寬鬆的相等概念。它只要求 **對稱性** ($a == b \implies b == a$) 和 **傳遞性** ($a == b \land b == c \implies a == c$)。
    
- **關鍵點**：它**不要求** 「反身性」 (Reflexivity)，也就是說，它**不保證** $a == a$ 總是為真。
    
- **最經典的例子**：`f32` 和 `f64` (浮點數)。
    
    - 在 IEEE 754 標準中，`NaN` (Not a Number) 被定義為 **不等於自己** (`NaN != NaN`)。
        
    - 因為存在這種「自己不等於自己」的情況，所以浮點數只能實現 `PartialEq`。
        

#### **`Eq` (完全相等 / 等價關係)**

- **定義**：這是更嚴格的相等。它繼承自 `PartialEq`，並且額外保證了 **反身性** ($a == a$ 永遠為真)。
    
- **實作**：`Eq` 實際上只是一個 **Marker Trait** (標記特徵)，它裡面沒有新的方法。只要你實作了 `Eq`，就代表你向編譯器保證：「這種類型的所有數值，都一定等於它自己」。
    
- **例子**：`i32`, `String`, `bool` 這些類型都實作了 `Eq`。
    
- **用途**：像 `HashMap` 的 Key 就要求必須是 `Eq`。因為如果 Key 不等於自己，Hash Table 就找不到它存進去的資料了（浮點數因此不能直接當 HashMap 的 Key）。
    

---

### 2. 順序性比較 (Ordering)

這兩個 Trait 負責處理 `<`, `>`, `<=`, `>=` 以及排序 (Sort)。

#### **`PartialOrd` (部分順序)**

- **定義**：允許兩個值進行比較，但 **結果可能是不存在的**。
    
- **返回值**：`partial_cmp` 方法返回的是 `Option<Ordering>`。
    
    - `Some(Less)`: 小於
        
    - `Some(Equal)`: 等於
        
    - `Some(Greater)`: 大於
        
    - `None`: 無法比較
        
- **例子**：又是浮點數。
    
    - 請問 `3.0` 和 `NaN` 誰大誰小？答案是無法比較。
        
    - 因此浮點數之間的比較可能返回 `None`。這也是為什麼你不能直接對包含浮點數的 `Vec` 使用 `.sort()` (因為 `.sort()` 需要確定的順序)，而必須用 `.sort_by(...)`。
        

#### **`Ord` (全序 / 總順序)**

- **定義**：集合中的 **任意** 兩個值都可以比較大小，絕對不會出現「無法比較」的情況。
    
- **返回值**：`cmp` 方法直接返回 `Ordering` (沒有 Option)。
    
- **關係**：要實作 `Ord`，該類型必須先實作 `Eq` 和 `PartialOrd`。
    
- **例子**：整數 `i32`、字串 `String`。它們任兩個值一定能分出高下。
    

---

### 總結對照表

| **Trait**        | **數學概念** | **核心特徵**      | **誰有實作?**                   | **誰沒有實作?**         |
| ---------------- | -------- | ------------- | --------------------------- | ------------------ |
| **`PartialEq`**  | 部分等價關係   | 支援 `==`, `!=` | 幾乎所有類型 (Int, Float, String) | -                  |
| **`Eq`**         | 等價關係     | 保證 `x == x`   | Int, String                 | **Float** (因為 NaN) |
| **`PartialOrd`** | 偏序關係     | 比較可能回傳 `None` | Int, Float, String          | -                  |
| **`Ord`**        | 全序關係     | 比較一定有結果       | Int, String                 | **Float** (因為 NaN) |

### 層級關係圖

Rust 的 Trait 有繼承關係，越下面的越嚴格：

1. **`PartialEq`** (最基本，有它才能比相等)
    
2. **`Eq`** (繼承自 `PartialEq`，保證反身性)
    
3. **`PartialOrd`** (依賴 `PartialEq`，允許 `< >` 運算)
    
4. **`Ord`** (繼承自 `Eq` + `PartialOrd`，保證嚴格排序)
    

### 寫程式時的建議

1. 自定義 struct/enum 時：
    
    通常你會希望盡可能加上 derive：
    
    Rust
    
    ```
    #[derive(Debug, PartialEq, Eq, PartialOrd, Ord)]
    struct MyId(u32);
    ```
    
2. 如果 struct 包含浮點數：
    
    你只能 derive PartialEq 和 PartialOrd。如果你強行 derive Eq 或 Ord，編譯器會報錯，因為浮點數不支援這些嚴格屬性。
    

這樣的設計雖然讓 Rust 在剛開始學的時候有點囉唆（為什麼我不能直接 sort 浮點數？），但它強制你在編譯階段就處理掉所有潛在的 NaN 比較錯誤，讓程式更安全。
